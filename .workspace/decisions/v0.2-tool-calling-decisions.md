# Decision: v0.2 Tool Calling Foundation

Date: 2026-06-27
Status: accepted

## Context

v0.2 makes tool calling a first-class runtime capability while keeping the
reducer pure and the runtime in control of the loop. Several design choices had
multiple valid options.

## Decision

1. Policy denial fails the run (`ToolCallRejected` -> `RunFailed`) rather than
   feeding the denial back to the LLM.
2. Retry uses a configurable backoff (`max_attempts`, `initial_delay`,
   `max_delay`, `backoff_multiplier`); v0.2 ships a simple geometric backoff.
3. The reducer decides retry deterministically from injected static config
   (`ReducerConfig`); the Tool Worker only classifies the error as retryable.
4. Tool retry reuses the `ScheduleTimer` action and `TimerFired` event with a
   `kind` field (`tool_retry` vs `run_timeout`) instead of a new action type.
5. Tool-call idempotency key is `runID:tool:toolCallID`, stable across attempts;
   dedup triggers only on a previously succeeded record.
6. `mock.fail_once` keys off the attempt number (1 fails, >=2 succeeds) rather
   than persisted per-key failure counting.
7. `internal/tool` is a standard-library-only leaf package so both `worker` and
   `runtime` can depend on it without an import cycle.
8. Integration/e2e tests use the real LLM (`orbis ws smoke tool`); no
   config-selectable mock LLM provider is added. Deterministic unit tests use
   in-package fakes.

## Rationale Summary

- Failing on denial is the simplest deterministic v0.2 behavior and keeps the
  event sequence clear; continuation needs loop-guard and context plumbing that
  belongs with v1 skill work.
- Keeping retry decisions in the reducer preserves determinism and testability;
  the worker has the typed error for classification only.
- Reusing the timer action avoids a parallel timer path and keeps run-timeout and
  retry-backoff on one mechanism.
- A stable idempotency key is what makes dedup and retry coherent; dedup only on
  success is what lets a failed attempt be retried.
- A leaf `tool` package is the cleanest way to share policy/retry types.

## Alternatives Considered

- Continuation on denial: more agentic but higher complexity and infinite-loop
  risk. Deferred to v1.
- New `ScheduleToolRetry` action: rejected; `ScheduleTimer` + `kind` is enough.
- Per-key failure counting for `fail_once`: rejected as less deterministic.
- Config-selectable mock LLM provider: rejected per the real-LLM testing
  preference; the real provider now supports tool calling.

## Consequences

- Easier: deterministic reducer tests, observable retries, safe-by-default tools.
- Harder: real-LLM tool tests are nondeterministic, so tool lifecycle is verified
  via the smoke path and via synthetic-event unit tests rather than asserting the
  real model's exact behavior.
