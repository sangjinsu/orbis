# Decision: v2.1 Learning Loop Hardening

Date: 2026-07-05
Status: accepted (recorded per feature as each lands)

## Context

v2 shipped the reviewable skill learning loop and explicitly deferred four
follow-ups: multi-version promotion, reviewer edits before approval,
session-independent lifecycle events, and real auth to replace the static
admin token. v2.1 implements them as four PRs, keeping the v2 invariants
(`SkillProposalCreated != SkillPromoted`, reducer purity, seed protection).

## Decisions

### §1 Multi-version promotion

1. **In-place bump, not multi-entry coexistence.** The index keeps one entry
   per id; promotion onto an existing learned id increments the version and
   replaces the body. A true multi-entry index would require reworking
   `parseIndex` duplicate-id rejection, the store's id keying, and file
   naming — none of which the read path needs.
2. **Seed protection via provenance, not tags.** An existing entry with an
   empty `source_proposal_id` is a curated seed and always conflicts
   (`ErrSkillConflict`). Provenance is written only by the promoter, so it is
   more robust than the editable `learned` tag.
3. **Integer-only learned versions; fail loudly on anything else.** Learned
   skills always carry `"1"`, `"2"`, ... A learned entry with a non-integer
   version is corrupt state; the bump errors (no silent fallback) and the
   proposal lands in the retryable `failed` state.
4. **Archive as `data/skills/archive/{id}@{old_version}.md`.** `@` cannot
   appear in a skill id, so archives never collide with active bodies; the
   loader only follows index paths, so the directory is inert.
5. **The v2 body-file-exists conflict check is removed.** It blocked the very
   retry the body-first-index-second write order was designed to allow: after
   a failed index write, the orphan `.md` made every re-promotion fail
   forever. Orphans are now documented as harmless and overwritten on retry.
6. **No new event types.** `SkillPromoted` gains an optional `version` payload
   field instead.

### §2 Reviewer edits before approval

1. **Structured fields only; body always re-rendered.** Reviewers edit the
   eight fields that compose the body; `Rerender()` re-derives body and hash
   through the creation renderer. Direct body edits could drift from the
   renderer's format and were rejected.
2. **Immutability by absence.** `skill_id`, `source_run_id`,
   `rationale_summary`, and `body` are not fields on the wire update type, so
   they are unsettable by construction rather than by validation.
3. **Pointer fields for partial update.** `nil` = keep, non-nil = replace —
   JSON absence vs. empty value is distinguished natively, unlike a
   fields-present map.
4. **`Revision` counter for uniqueness.** Repeated edits need unique audit and
   event ids under the fixed-clock test model; a per-proposal revision counter
   is deterministic where timestamps are not.
5. **PATCH verb.** The repo already uses Go 1.22 method patterns; PATCH states
   the partial-update semantics precisely.
6. **The Get → mutate → Update race is accepted** for the single-admin review
   model and documented in code instead of adding store-level compare-and-swap.
