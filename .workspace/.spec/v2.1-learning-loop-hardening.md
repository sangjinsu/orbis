# v2.1 Learning Loop Hardening

## Status

In progress. Four features hardening the v2 reviewable skill learning loop:
multi-version promotion (§1), reviewer edits before approval (§2),
session-independent lifecycle events (§3), named multi-token auth (§4).
Sections are added as each lands.

The v2 invariants are unchanged throughout: `SkillProposalCreated !=
SkillPromoted` (no pending→promoted transition), the reducer stays pure (all
learning-loop I/O lives in the app layer), and curated seed skills are never
replaced by the loop.

## §1 Multi-version promotion (in-place bump)

### Behavior

`Promoter.Promote` no longer rejects every existing skill id. When the target
id is already in the index:

- **Learned entry** (`source_proposal_id` non-empty): the promotion bumps the
  entry in place. The version increments as an integer (`"1"` → `"2"`, via
  `nextVersion`), the previous body is archived, the new body replaces
  `{skill_id}.md`, and the index entry refreshes its content-bearing fields.
- **Seed entry** (`source_proposal_id` empty): `ErrSkillConflict`, exactly as
  v2. Curated seeds are never replaced; provenance is the discriminator
  because it is harder to forge than a tag.
- **Learned entry with a non-integer version** (e.g. `"1.0.0"`): the bump
  fails loudly. Learned versions are always plain integers, so anything else
  marks a corrupt entry; the proposal lands in the retryable `failed` state
  through the existing `failPromotion` path.

### Index entry on bump

Refreshed: `title`, `description` (= purpose), `version`, `related_tools`,
`source_proposal_id`, `source_run_id`, `created_at`. Preserved: `id`, `name`,
`path`, `tags`, `triggers`, `status`, `priority`, and the entry's position in
the index. The index keeps one entry per id, so the store, selector, and
`parseIndex` (duplicate-id rejection) are untouched.

### Archive

```
data/skills/archive/{skill_id}@{old_version}.md
```

`@` is outside the `validSkillID` charset, so an archived file can never
collide with an active skill body. The loader only reads paths listed in the
index, so `archive/` is inert. Re-archiving the same version overwrites
identical content; a missing body (orphaned index entry) archives nothing.

### Write order and failure

Validate + compute version → archive old body → write new body → rewrite
index. The v2 body-first-index-second rationale holds: a failed index write
leaves an orphan `.md` while the index (the load source of truth) stays
intact. The v2 body-file-exists conflict check is **removed** — it made that
very retry impossible (the orphan blocked re-promotion forever); orphans are
now overwritten by the retry. A re-promotion retry after a partial failure may
bump past the intended version (audit preserves the history).

### Events

No new event types. `skillLearningEventPayload` gains `version` (omitempty),
filled on `SkillPromoted`, so feed consumers see which version a promotion
produced.

### Removed

`EnsureSkillIDAvailable` — production code never used it and "available"
became a false notion once existing learned ids are legal targets.

## §2 Reviewer edits before approval (structured fields only)

### Behavior

A pending proposal can be edited by an admin before the approve/reject
decision, via WS `skill.proposal.update` or HTTP
`PATCH /skill-proposals/{proposalID}` (both admin-gated).

- **Editable** (the eight fields that compose the rendered body): `title`,
  `purpose`, `when_to_use`, `required_context`, `procedure`, `related_tools`,
  `verification`, `pitfalls`.
- **Not editable**: `skill_id`, `source_run_id` (identity), `rationale_summary`
  (detection record, not content), `body` (always re-rendered) — these fields
  simply do not exist on the wire request type
  (`protocol.SkillProposalUpdateRequest`), so they cannot be set.
- Partial update via pointer fields: `nil` = unchanged, non-nil = replace.
  Lists may be cleared with `[]` (their body section disappears); `title` may
  not be emptied. At least one field must be provided.
- After applying the edit, `SkillProposal.Rerender()` re-derives `Body` and
  `ContentHash` through the same renderer used at creation, so an edited
  proposal can never drift from what promotion writes.
- Only `pending` proposals are editable; the status does not change on edit.

### Revision counter

`SkillProposal.Revision` (0 = as created) increments on each edit and keys the
per-edit audit id (`audit_{proposal}_{event}_r{revision}`) and event id
(`{source_run}:skill_proposal_updated:{revision}`), keeping repeated edits
unique and deterministic under a fixed clock. `Revision` is exposed on the
wire summary.

### Events and audit

`SkillProposalUpdated` → `SkillAuditRecorded`, on the source run's session
(the approve/reject pattern). The audit summary names the edited fields
(`"proposal updated: title, procedure"`); payloads stay metadata-only.

### Concurrency note

The service's Get → mutate → Update sequence is not atomic across callers;
the single-admin review model accepts this (documented in code). The store
still validates and enforces transitions on every write.

## §3 Session-independent lifecycle events (global feed)

### Behavior

The broker gains a global subscriber set: `SubscribeGlobal(ctx)` /
`PublishGlobal(event)`. `Publish` stays session-only — the broker knows no
event policy (dumb pipe). The app layer owns the whitelist:
`RuntimeService.publish` fans the same sequenced wire event out to global
subscribers when `isSkillLifecycleEvent(type)` — the eleven skill-learning
events (v2's ten plus `SkillProposalUpdated`). Per-run events stay
session-only.

Clients subscribe with the existing `session.subscribe` method and a new
`scope` param: `"global"` needs no `session_id`; empty scope keeps the v0.1
behavior (session subscription, `session_id` required); any other value is an
error.

### Standalone reload is no longer silent

`RuntimeService.ReloadSkills` (behind both `POST /skills/reload` and WS
`skill.reload`) emits `SkillIndexReloadRequested` before reloading and
`SkillIndexReloaded` on success — global-only events with no `session_id`, no
`seq`, payload `{actor, count}` (actor is `"admin"` until named principals in
§4). The approve flow calls `s.skills.Reload()` directly and emits its own
session-scoped events (which also fan out globally), so nothing double-emits.
With skills disabled the reload errors before emitting anything.

### Delivery guarantees

Global delivery matches session subscriptions: buffered (16), non-blocking,
drops on overflow. Lifecycle events remain persisted only on the source run's
session; the global feed is a live stream and recovery goes through
`GET /skill-proposals`. The feed is read-open like every read surface in the
codebase — a conscious decision (payloads are metadata-only).
