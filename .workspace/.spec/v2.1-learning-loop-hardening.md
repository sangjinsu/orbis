# v2.1 Learning Loop Hardening

## Status

In progress. Four features hardening the v2 reviewable skill learning loop:
multi-version promotion (§1), reviewer edits before approval (§2),
session-independent lifecycle events (§3), named multi-token auth (§4).
Sections are added as each lands.

The v2 invariants are unchanged throughout: `SkillProposalCreated !=
SkillPromoted` (no pending→promoted transition), the reducer stays pure (all
learning-loop I/O lives in the app layer), and curated seed skills are never
replaced by the loop.

## §1 Multi-version promotion (in-place bump)

### Behavior

`Promoter.Promote` no longer rejects every existing skill id. When the target
id is already in the index:

- **Learned entry** (`source_proposal_id` non-empty): the promotion bumps the
  entry in place. The version increments as an integer (`"1"` → `"2"`, via
  `nextVersion`), the previous body is archived, the new body replaces
  `{skill_id}.md`, and the index entry refreshes its content-bearing fields.
- **Seed entry** (`source_proposal_id` empty): `ErrSkillConflict`, exactly as
  v2. Curated seeds are never replaced; provenance is the discriminator
  because it is harder to forge than a tag.
- **Learned entry with a non-integer version** (e.g. `"1.0.0"`): the bump
  fails loudly. Learned versions are always plain integers, so anything else
  marks a corrupt entry; the proposal lands in the retryable `failed` state
  through the existing `failPromotion` path.

### Index entry on bump

Refreshed: `title`, `description` (= purpose), `version`, `related_tools`,
`source_proposal_id`, `source_run_id`, `created_at`. Preserved: `id`, `name`,
`path`, `tags`, `triggers`, `status`, `priority`, and the entry's position in
the index. The index keeps one entry per id, so the store, selector, and
`parseIndex` (duplicate-id rejection) are untouched.

### Archive

```
data/skills/archive/{skill_id}@{old_version}.md
```

`@` is outside the `validSkillID` charset, so an archived file can never
collide with an active skill body. The loader only reads paths listed in the
index, so `archive/` is inert. Re-archiving the same version overwrites
identical content; a missing body (orphaned index entry) archives nothing.

### Write order and failure

Validate + compute version → archive old body → write new body → rewrite
index. The v2 body-first-index-second rationale holds: a failed index write
leaves an orphan `.md` while the index (the load source of truth) stays
intact. The v2 body-file-exists conflict check is **removed** — it made that
very retry impossible (the orphan blocked re-promotion forever); orphans are
now overwritten by the retry. A re-promotion retry after a partial failure may
bump past the intended version (audit preserves the history).

### Events

No new event types. `skillLearningEventPayload` gains `version` (omitempty),
filled on `SkillPromoted`, so feed consumers see which version a promotion
produced.

### Removed

`EnsureSkillIDAvailable` — production code never used it and "available"
became a false notion once existing learned ids are legal targets.
