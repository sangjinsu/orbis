# v0.2 Tool Calling

## Status

Accepted

## Purpose

Make tool calling a first-class runtime capability. The LLM may only propose
tool calls; the runtime validates, authorizes, dispatches, executes, observes,
and persists them as events. The reducer owns control flow; only the Tool Worker
executes tools.

## Scope

- `internal/tool`: `Tool` interface, schema/metadata, registry, policy, retry
  policy, idempotency helpers, mock tools.
- `internal/worker/tool_worker.go`: policy → dedup → timeout → execute → persist.
- `internal/store/tool_call_store.go`: `data/tool_calls/{sanitized_key}.json`.
- `internal/runtime`: reducer retry/reject/timer logic, dispatcher tool
  lifecycle events + retry timer, `context_builder.go` for LLM context.
- `internal/worker/llm_provider.go`: send tool schemas, parse `function_call`,
  thread conversation context.

## Non-Goals

Skill store/selection/auto-creation, tool search, subagents, human approval UI,
multi-channel gateway, MCP, K8s worker separation, distributed broker, prod DB,
and a config-selectable mock LLM provider (tests use the real LLM or in-package
fakes).

## Data Model

### Tool (`internal/tool`)

- `Tool`: `Name`, `Description`, `Schema`, `Metadata`, `Execute`.
- `ToolMetadata`: `SideEffectLevel`, `Toolset`, `Timeout`, `RetryPolicy`,
  `RequiresApproval`, `IdempotencyRequired`.
- `SideEffectLevel`: none, read, write, network, dangerous.
- `Toolset`: safe, read, write, network, runtime, dangerous. Default enabled: safe.
- `RetryPolicy`: `MaxAttempts`, `InitialDelay`, `MaxDelay`, `BackoffMultiplier`,
  `RetryableErrors`.

### Tool call record (`data/tool_calls/{key}.json`)

`idempotency_key`, `session_id`, `run_id`, `tool_call_id`, `tool_name`,
`status` (running/succeeded/failed), `attempts`, `result`, `error`,
`created_at`, `updated_at`. The key is sanitized to a filesystem-safe token with
a short hash suffix so distinct keys never collide.

### Events

`ToolCallStarted`, `ToolCallSucceeded`, `ToolCallFailed`, `ToolCallRejected`,
`ToolCallTimedOut`, `ToolCallRetryScheduled`, `ToolCallRetried`,
`ToolCallDeduplicated`. (`ToolCallProposed` is defined but not emitted; the
proposal is carried by `LLMResponseReceived`.) Each tool event payload may carry
`tool_call_id`, `tool_name`, `idempotency_key`, `attempt`, `max_attempts`,
`duration_ms`, `result`, `error`, `reason_code`, `retryable`.

### Actions

`DispatchToolCall` (tool_call_id, name, args, idempotency_key, attempt,
max_attempts, timeout) and `ScheduleTimer` (kind, delay, tool_call) reused for
retry backoff.

## Flow

```
LLMResponseReceived (tool_call)
  -> reducer: WAITING_TOOL, record assistant tool-call turn, DispatchToolCall (attempt 1)
  -> dispatcher: ToolCallStarted -> Tool Worker
       Tool Worker: policy -> dedup -> context.WithTimeout(execute) -> persist
  -> success    : ToolCallSucceeded -> reducer dispatches next LLM call
  -> dedup      : ToolCallDeduplicated + ToolCallSucceeded (cached result)
  -> rejected   : ToolCallRejected -> reducer fails run
  -> timeout    : ToolCallTimedOut  \ reducer: retry (WAITING_TIMER + ScheduleTimer)
  -> failed     : ToolCallFailed    / or fail run when attempts exhausted
       retry: ScheduleTimer(tool_retry) -> TimerFired(tool_retry)
              -> reducer: ToolCallRetried + DispatchToolCall(attempt+1)
```

## Policy

`Check` order (reason codes): `unknown_tool`, `toolset_not_allowed`,
`side_effect_denied`, `missing_idempotency_key`, `approval_required`,
`timeout_exceeds_max`, `invalid_args`. Dangerous tools are denied by default
(blocked by both toolset and side-effect level). Denial emits `ToolCallRejected`
and fails the run.

## Idempotency

Tool action key is `runID:tool:toolCallID`, stable across retries. The Tool
Worker deduplicates only when a prior record has status `succeeded`; `running`
and `failed` records still execute so retries work. `fail_once` keys off the
attempt number (1 fails, ≥2 succeeds).

## Configuration

`ORBIS_TOOLSETS` (default safe), `ORBIS_TOOL_TIMEOUT_DEFAULT` (5s),
`ORBIS_TOOL_TIMEOUT_MAX` (30s), `ORBIS_TOOL_RETRY_MAX_ATTEMPTS` (2),
`ORBIS_TOOL_RETRY_INITIAL_DELAY` (500ms), `ORBIS_TOOL_RETRY_MAX_DELAY` (5s),
`ORBIS_TOOL_RETRY_BACKOFF` (2.0). Defaults are safe; dangerous tools never enabled
implicitly.

## Real LLM Tool Calling

The OpenAI Responses provider advertises tool schemas as flattened function
definitions (`{type:function,name,description,parameters}`), rebuilds the
conversation as input items (`function_call` / `function_call_output`), and
parses `function_call` output items into a proposed tool call. Tool-only
responses (no text) are valid.

## Edge Cases

- Cancelled runs drop later tool transitions (reducer cancellation guard) and the
  retry timer aborts on context cancellation.
- Run timeout (`TimerFired kind=run_timeout`) still fails the run regardless of
  tool state.
- Tool result bytes are compacted on dedup re-emit for consistency with a fresh
  execution.

## Testing Requirements

Deterministic reducer/dispatcher unit tests (synthetic events, in-package fake
provider), Tool Worker tests (success/dedup/timeout/rejection/fail-once),
provider tool-schema + function_call tests, and a runtime end-to-end retry test.
Manual/e2e uses the real LLM via `orbis ws smoke tool`.

## Open Questions

- Should policy denial feed the reason back to the LLM (continuation) instead of
  failing the run? Deferred to v1.
- Concurrent same-key execution (status `running`) is not blocked; the session
  lane makes this rare within a run.
